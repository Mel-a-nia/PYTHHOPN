
## 1. Цель работы

В этой лабораторной я научился строить иерархию классов на основе уже существующего класса из ЛР-1. Главная идея — не повторять код, а наследовать то что уже есть и добавлять только новое. Также я разобралac с полиморфизмом — когда разные объекты отвечают по-разному на один и тот же вызов.

## 2. Описание реализованной иерархии классов

### Базовый класс —> Player
Это тот же Playerd из ЛР-1. Я его не переписывалa, просто взял готовый и добавил два общих метода которых не было раньше.
Первый метод возвращает текстовое описание Player. Каждый тип игрока описывает себя по-своему. Второй метод — общая атака. Базовый игрок атакует просто, маг(mago) бросает заклинание, лучник (Arqueroo) стреляет стрелой. Главное что метод называется одинаково для всех — можно вызвать его не зная заранее какого типа игрок перед тобой.

### Дочерний класс —> Mago (Маг)
Наследует всё от базового Player и добавляет своё.
Новые характеристики: ресурс для заклинаний который тратится при каждой атаке, и тип магии которую использует маг — огонь, лёд или молния.
Новые возможности: маг умеет бросать заклинания тратя свой ресурс, и атакует именно через них.
Изменённое поведение по сравнению с базовым игроком: маг получает вдвое больше опыта чем обычный игрок, его текстовое описание и вывод информации показывают его ресурс и тип магии, и его атака — это всегда заклинание.

### Дочерний класс —> Arquero (Лучник)
Наследует всё от базового игрока и добавляет своё.
Новые характеристики: количество стрел которые можно потратить, и показатель скорости который помогает уклоняться от урона.
Новые возможности: лучник умеет стрелять стрелами уменьшая их запас, и атакует именно через стрельбу.
Изменённое поведение по сравнению с базовым игроком: лучник получает вдвое меньше урона благодаря скорости, его текстовое описание и вывод информации показывают стрелы и скорость, и его атака — это всегда выстрел.

### Иерархия
Базовый Player из первой лабораторной стоит на вершине. От него наследуют Маг и Лучник — они берут всё что у него есть и добавляют своё сверху.

Создаются три типа объектов: обычный Player, два мага (Magos) и два лучника (Arqueros). Для каждого выводится информация и текстовое описание — каждый тип отвечает по-своему. Также проверяется принадлежность объектов к типам.

## 4. Вывод

В этой лабораторной я изучил наследование и полиморфизм.
Наследование позволяет не переписывать уже готовый код. Дочерний класс получает всё от родителя автоматически — валидацию, свойства, состояния — и просто добавляет сверху то что ему нужно.
Полиморфизм позволяет вызвать один и тот же метод для разных типов объектов и получить разный результат. Не нужно писать условия проверки типа — каждый объект сам знает что делать когда его вызывают.
Фильтрация по типу позволяет из одной общей коллекции получить только нужные объекты, что удобно когда в списке хранятся объекты разных типов одновременно.
